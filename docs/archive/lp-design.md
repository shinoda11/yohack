# YOHACK LP設計書 v1.0

## 1. 前提条件

### ビジネス前提
- **販売開始**: 2025年7月以降
- **価格**: ¥29,800 / 90日（Pass モデル踏襲）
- **運営者稼働**: ゼロで回る設計（1on1は裏メニュー、導線には入れない）
- **炎上回避**: 絶対条件。アカウントは個人名ではなく「YOHACK」名義
- **隠す戦略**: 一見さんお断りの思想を維持

### ターゲット
- 世帯年収 1,000〜3,000万円の DINKs / プレ DINKs
- 都市部在住、28〜40歳
- 7,000〜10,000万円クラスの住宅購入を検討中
- 「買えるかどうか」ではなく「買った後も動けるかどうか」が論点

### 流入経路
```
Instagram（問い × ケース投稿）
  ↓  「自分のことだ」と思った人だけがリンクを踏む
LP（本ドキュメントの対象）
  ↓  「答えを出したい」と思った人だけが進む
FitGate（12問・自動判定）
  ├─ Prep → レター登録（再診断リンク付き）
  └─ Ready → 決済画面が開示される → Pass購入
       ↓
YOHACK本体（セルフサービス）
```

### Instagram が LP の前段で果たす役割
Instagram 投稿は LP の「第0層フィルター」。投稿内の数字レンジ（世帯年収 1,500〜3,000万、物件 7,000〜10,000万）がそのままターゲット選別になっている。LP に来る人は「すでに問いに引っかかった人」であり、ゼロから説明する必要がない。

## 2. LPの役割定義

### LPの唯一の仕事
**「FitGate に進みたい」と思わせること。**

売るのは LP の仕事ではない。FitGate 通過後の決済画面の仕事。LP は「この先に進む価値がある」と感じさせるところまで。

### 見せるもの / 隠すもの

| 見せる（LP上で公開） | 隠す（FitGate通過後に初めて見える） |
|---|---|
| 問い（Q01〜Q20の一部） | YOHACK の実際の UI 画面 |
| ケースの前提条件と結論 | 計算ロジック・シミュレーションの中身 |
| 「世界線比較」という概念 | 世界線比較の具体的な操作方法 |
| 「余白」「安心ライン」の考え方 | 余白スコアの算出ロジック |
| フラットな立場の宣言 | 価格（¥29,800） |
| 対象 / 非対象の条件 | 決済 URL |
| チラ見せ動画（15秒、数字ぼかし） | フルスクリーンショット |

**価格を隠す理由**: 価格が見えると「¥29,800の価値があるか」という損得計算が始まる。価格が見えないと「自分が対象かどうか」という適合判断になる。後者の方がFitGateへの遷移率が高い。

## 3. セクション構成（7セクション）

### S1: ヒーロー — 問いの再提示

**役割**: Instagram で見た問いの「続き」として機能させる。

**コピー案**:

> この家を買ったあと、年収が 20% 下がっても、まだ動けるか。
>
> この問いに、数字で答えられますか。
>
> YOHACK は、人生の選択肢を「世界線」として並べ、
> 安心ラインと余白を見える化する意思決定 OS です。

**ビジュアル要素**:
- メインビジュアル: 15秒チラ見せ動画（後述）
- または問い辞書から3〜4本を縦に並べて「これ、答えられますか？」と問いかける

**CTA**: 「あなたのケースで確認する」→ FitGate へ

**チラ見せ動画の仕様**:
- 内容: ダッシュボードで年収スライダーを動かすと、資産推移チャートの3本線とスコアがリアルタイムで変わる様子
- 数字: ぼかす or ダミーデータ（実際の計算ロジックは推測できない程度に）
- 長さ: 15秒、ループ再生、音声なし
- 世界線比較の表は見せない（Pro機能の核心なので隠す）

---

### S2: 何が見えるようになるか — 成果物の提示

**役割**: 「使うと何が分かるか」を結論だけ見せる。過程は見せない。

**構成**: ケース台帳 C01〜C04 から 2〜3本を使う。

**ケース例 1（C01ベース）**:
> **世帯年収 2,400万・家賃 32万の夫婦**
> 外資 IT ＋メガバンク。都心 8,500万の 2LDK を検討中。海外転職も視野。
>
> 世界線比較の結果:
> 「家を買う / 買わない」より「駐在を取るかどうか」の方が、全体の余白に効いていた。
>
> → 詳細は YOHACK 本体で確認できます。

**ケース例 2（C02ベース）**:
> **世帯年収 2,400万・コンサル × 事業会社**
> ペースダウンしたい気持ちはあるが、今のうちに買うべきか迷っている。
>
> 世界線比較の結果:
> 年収 1,800万 → 1,200万のペースダウンを織り込んでも、8,000万ラインなら安心ラインを割らなかった。
>
> → 詳細は YOHACK 本体で確認できます。

**設計ポイント**:
- 数字の前提（年収・物件価格）は具体的に出す → ターゲットが「自分に近い」と判断する材料
- 結論（「〇〇の方が効いていた」「安心ラインを割らなかった」）は出す → 「こういう答えが出るのか」という期待を作る
- 過程（どの画面でどう操作して、どの計算を経てその結論に至ったか）は出さない → 隠す戦略の核

---

### S3: 3つの軸 — 世界線・安心ライン・余白

**役割**: YOHACK の思考フレームワークを簡潔に伝える。

**コピー案**（各1文で完結）:

> **世界線**: 同じ前提で、異なる選択肢を並べて比較する。
> 「家を買う世界線」と「買わない世界線」を、100歳まで並走させる。
>
> **安心ライン**: 「ここを割ると厳しい」の下限を見える化する。
> 生存率と資産推移から、あなたの安心ラインを算出する。
>
> **余白**: その差分を「どの年代にどれだけ余裕が残るか」に翻訳する。
> お金・時間・リスクの3軸で、人生の余白を定量化する。

**ビジュアル**: 3つのカードUI（既存LPのS3を簡略化して転用）

---

### S4: フラットな立場 — 何も売らない宣言

**役割**: 信頼の根幹。既存LPからほぼそのまま転用。

**コピー案**:

> YOHACK は、物件も保険も投資商品も売りません。
>
> どの世界線を選んでも、この OS の利益は変わりません。
>
> 返すのは結論ではなく、比較と判断の土台です。

**設計ポイント**:
- 短くていい。3行で十分。
- Instagram で繰り返し「問い」だけを投げて「答え」を売らないスタンスを取っているので、LP では確認するだけで信頼が成立する。

---

### S5: 向いている人 / 向いていない人

**役割**: 自己選別を促す。「向いていない人」の方が重要。

**向いている人**:
- 世帯年収 1,000〜3,000万で、7,000〜10,000万クラスの物件を本気で検討している
- 「買えるか」ではなく「買った後も動けるか」を気にしている
- 夫婦で論点を整理するための客観データが欲しい
- 自分で数字を動かして判断したい（人に答えを出してほしいのではなく）

**向いていない人**:
- 予算 5,000万以下 or 1.5億以上（レンジ外）
- 「買うか買わないか」をすでに決めている（比較が不要）
- 数字よりもフィーリングで決めたい
- ファイナンシャルプランナーに全部任せたい

**設計ポイント**:
- 「向いていない人」を明確にすることで、ターゲットは「自分は向いている側だ」と確信する
- 一見さんお断りの思想そのもの。排除すること自体が信頼シグナル

### S6: FAQ（4問）

**Q1: 何をしてくれるサービスですか？**
A: 人生の選択肢（住宅購入・キャリア変更・家族計画など）をシミュレーションし、「世界線」として並べて比較するツールです。モンテカルロシミュレーションで 100歳までの資産推移を計算します。物件紹介や投資助言は行いません。

**Q2: 物件の紹介や投資のアドバイスはありますか？**
A: ありません。YOHACK はどの選択肢を選ぶべきかを提案しません。比較に必要な数字の土台を返すだけです。

**Q3: 個人情報はどう扱われますか？**
A: シミュレーションに使用する情報は、セッション内でのみ使用されます。第三者への提供は行いません。

**Q4: 適合チェックの後に何が起きますか？**
A: 12問のチェックに回答いただくと、自動で判定結果が出ます。条件に合う方には、YOHACK へのアクセス方法をご案内します。条件に合わない場合も、その旨をお伝えします。

---

### S7: CTA — 適合チェックへ

**コピー案**:

> あなたのケースで、世界線比較が意味を持つかどうか。
> 12問のチェックで確認できます。
>
> [適合チェックに進む]

**設計ポイント**:
- 「購入」「申し込み」という言葉は使わない
- 「確認する」「チェックする」というニュートラルな動詞
- FitGate の先に何があるかは敢えて明示しない（価格を隠す戦略と整合）

## 6. FitGate の役割拡張

### 3つの役割を同時に担う

**役割A: 冷やかし排除**
12問答えるだけで冷やかしの大半は脱落。

**役割B: アイデア保護**
FitGate を通過しないと YOHACK の UI・機能・計算ロジックが一切見えない。

**役割C: オンボーディングデータ収集**
12問の回答でプロファイルを自動プリセット。Pass 購入後に「初期設定済みのシミュレーション」が表示される。

### FitGate → YOHACK プリセット対応表

| FitGate の質問 | YOHACK Profile フィールド |
|---|---|
| 個人年収レンジ | `grossIncome` |
| 世帯年収レンジ | `grossIncome + partnerGrossIncome` |
| 年齢 | `currentAge` |
| 家族構成 | `mode` (solo/couple) |
| 現在の家賃 | `housingCostAnnual`（月額→年額変換） |
| 検討物件価格帯 | 住宅プラン初期値 |
| 貯蓄＋投資 | `assetCash + assetInvest`（按分は仮置き） |

### Google Forms → FitGate 統合
現在の Google Forms の質問（メールアドレス、居住エリア、年収、家賃、物件価格帯、モヤモヤ自由記述）を FitGate に吸収する。特に「今、家と Exit についていちばんモヤモヤしていること」の自由記述は、コンテンツのネタの宝庫 + ユーザー理解に直結するため、FitGate の最終問として追加。

## 7. 技術スタック判断

### 推奨: 案A（既存LP改修）→ 段階的に案Bへ統合

**フェーズ1（〜5月）: 既存 LP を改修**
- Vite + Express + MySQL の既存基盤を活用
- FitGate, Stripe 決済, メール送信は既に動いている
- DB スキーマ 12テーブル整備済み
- セクション構成を本ドキュメントに沿って書き直す
- チラ見せ動画を撮影・編集して埋め込む

**フェーズ2（5〜6月）: YOHACK 本体に認証を入れる**
- Supabase 導入（認証 + データ永続化）
- localStorage → Supabase DB への移行
- FitGate 回答データ → Supabase → YOHACK プロファイル自動プリセットのパイプライン

**フェーズ3（6〜7月）: 導線の全接続**
- LP（FitGate 通過）→ Stripe 決済 → YOHACK 認証付きアクセスの一気通貫
- プロダクト URL のアクセス制御（認証なしでは入れない状態）
- 7月に「売れる状態」で開通

### 既存 LP リポジトリの活用資産
- `fitGateResponses` テーブル + 12問のフォームUI
- `passSubscriptions` テーブル + Stripe Checkout 連携
- `invitationTokens` テーブル（将来の Session 裏メニュー用）
- DM テンプレート（M4 シートから転用）
- メール送信基盤（SendGrid）

## 9. 炎上回避の構造設計

1. **Instagram アカウントは「YOHACK」名義で運用。** 個人名・顔・会社名は出さない。
2. **「煽り」を排除し「問い」に徹する。** 「家を買うな！」ではなく「年収が 20% 下がっても動けるか？」
3. **金額を「バカにする」文脈を作らない。** 「年収 1,500万でも買えない」ではなく「どの年代に余白を置くか」。
4. **LP に「ファイナンシャルアドバイスではない」旨の disclaimerを明記。** 既存 LP の S9（モデルの前提と制約）をフッターに簡易版で残す。
